## Overview

Apollo.io uses a credit-based pricing model where users spend credits to access contact data (like emails, phone numbers, and exports), with higher plans offering more monthly credits.

### Purchase credits vs Plan-based credits

- **Purchase credits (direct):** The customer buys credits with money without a subscription. You top up their wallet; there is no plan or entitlement. Use this for one-off credit packs or add-ons.
- **Plan-based credits:** The customer subscribes to a plan and receives **recurring credit grants** (e.g. 30,000 credits per month) plus **entitlement** to specific features. Credits are tied to the subscription; when the plan is attached, grants and feature access are provisioned together.

You can offer one path, the other, or both (e.g. subscription with monthly grants plus the option to buy extra credits).

## 1. Key Chargeable Features in Apollo

- Apollo uses a prepaid credit burndown model for contact data access. Credits are deducted immediately when a user performs a data-reveal or enrichment action.

### A. Contact Data Unlock & Enrichment

#### How Credits Are Applied

**Prepaid Allocation**: When a user subscribes **allocated_credits = plan_credit_limit**. Plans can be billed annually with **recurring grants** (e.g. credits per month).
- **Professional plan → 48,000 credits/year** (or e.g. 4,000/month as recurring grants)
- **Basic → 30,000 credits/year** (or e.g. 2,500/month as recurring grants)

Example: Basic plan, **annually billed**, includes a set number of credits per month (recurring grants) for outbound/reach usage.

**Action-Based Deduction**

| Action                | Credit Cost          |
| --------------------- | -------------------- |
| Unlock verified email | 1 credit             |
| Unlock mobile number  | 5–10 credits         |
| Export contact        | 1 credit             |
| API enrichment        | 1 credit per record  |


### B. Email Sending & AI Outreach Credits

#### How Credits Are Applied

**Email Sending Model** Depending on plan:
- **Daily send limits per mailbox**
- **OR credits per email sent**

| Action                       | Credit Cost         |
| ---------------------------- | ------------------- |
| Send 1 email                 | 1 credit            |
| AI-generated personalization | 1 additional credit |
| Email verification           | 1 credit per email  |




## 2. Apollo-Style Credit Model

### Key Behaviors

<Steps>
  <Step title="Purchase credits">
    Users **purchase credits** (packages or add-ons).
  </Step>
  <Step title="Set cost per action">
    Each **action has a credit cost** (e.g. 1 for search, 5 for export).
  </Step>
  <Step title="Deduct and track">
    Credits are **deducted per action**; balance is tracked per customer.
  </Step>
  <Step title="Enforce balance">
    When balance is **zero**, block the action or prompt to buy more.
  </Step>
</Steps>

### Example: Credit Cost per Action

| Action            | Credit Cost |
| ----------------- | ----------- |
| Contact Search    | 1 credit    |
| Contact Export    | 5 credits   |
| Email Enrichment  | 2 credits   |
| Phone Reveal      | 10 credits  |

### Example: Credit Packages (Prepaid)

| Plan    | Credits  | Price  |
| ------- | -------- | ------ |
| Starter | 1,000    | $29    |
| Growth  | 5,000    | $99    |
| Pro     | 20,000   | $299   |

Credits can reset monthly, never expire, or expire after X days—depending on your product.

**Overage:** After included credits are used, charge overage (e.g. 1.00 USD per credit or per use). Flexprice can bill usage above the grant; configure overage on the plan or charge.

## 3. When to Use Credit-Based Pricing

<Steps>
  <Step title="Different cost weights">
    Different actions have **different cost weights**.
  </Step>
  <Step title="Variable usage">
    Usage **varies a lot per customer**.
  </Step>
  <Step title="Flexible consumption">
    You want **flexible consumption** and add-ons or top-ups.
  </Step>
</Steps>

## 4. Complete Flow Summary

The end-to-end flow for credit-based pricing in Flexprice:

<Steps>
  <Step title="Create Price Unit">
    Create a **Price Unit** (e.g. “Credits”) with conversion to base currency.
  </Step>
  <Step title="Create Features">
    Create **metered features** for each action that costs credits (e.g. Contact Export, Email Enrichment). Link them to plans via **entitlements** so you can control which features a customer can use and optional free usage (e.g. 50 free then deduct from credits). See [Linking features to plans](/docs/product-catalogue/features/linking-to-plans).
  </Step>
  <Step title="Create Plan">
    Create a **Plan** that includes credit packages or usage charges (e.g. annually billed with recurring credit grants per month).
  </Step>
  <Step title="Create Customer">
    Create **Customer** (with External ID for your system).
  </Step>
  <Step title="Create Wallet in Credits">
    Create **Wallet** in the Credits Price Unit for that customer.
  </Step>
  <Step title="Credit grant and entitlement">
    For **plan-based credits:** Attach a **subscription** (customer + plan). Credits are granted per the plan (e.g. recurring monthly grants). **Entitlement** (which features the customer can use, and any free tier) comes from the plan. For **direct purchase** only, skip subscription and top up the wallet instead.
  </Step>
  <Step title="Top up wallet">
    **Top up** the wallet (initial purchase, add-ons, or one-off credits). For plan-based, grants may fill the wallet automatically when the subscription is active.
  </Step>
  <Step title="On each action: Check entitlement → balance → debit">
    When the user performs an action: (1) **Check entitlement**—does this customer’s plan allow this feature? (2) **Check wallet balance**—do they have enough credits? (3) If yes, **debit** the wallet (via API or Manual Debit) and perform the action; if no, block or prompt to buy more.
  </Step>
  <Step title="Optional: Auto top-up">
    Configure **auto top-up** when balance is low.
  </Step>
  <Step title="Billing / Invoices">
    Flexprice bills in base currency; wallet balance is shown in credits.
  </Step>
</Steps>

## Flexprice Set-up

To implement Apollo-style credit-based pricing in Flexprice, follow these steps in order.

<Steps>

  <Step title="Step 1: Create Features (Metered)">
    Navigate to **Flexprice Dashboard** → **Product Catalog** → **Features**

     <Frame>
  ![Create Features (metered) for credit-based pricing](/public/images/docs/subscription-flow/credit-based-features.png)
  </Frame>

    Click **Create Feature** for each action that costs credits (e.g. Contact Export, Email Enrichment). Use **Metered** feature type; set **Event Name** and **Aggregation** so usage events can be sent. You will link these features to your plan and set **entitlements** (e.g. free usage limit, then deduct from credits).

    For more, check-out [Creating a feature](/docs/product-catalogue/features/create) and [Linking features to plans](/docs/product-catalogue/features/linking-to-plans).
  </Step>

  <Step title="Step 2: Create Plan (with Credits or Recurring Grants)">
    Navigate to **Flexprice Dashboard** → **Product Catalog** → **Plans**

    Click **Add** to create a plan. Configure plan details (e.g. **Basic**, annually billed). Add charges that deliver credits (e.g. recurring credit grant per month) or link metered features with pricing. Configure **entitlements** for the plan so subscribers get access to the right features (and optional free tier before credits are deducted).

    For plan-based credits, when a customer subscribes to this plan, they receive the configured credit grants (e.g. monthly). For more, check-out [Creating a Plan](/docs/product-catalogue/plans/create) and [Linking features to plans](/docs/product-catalogue/features/linking-to-plans).
  </Step>

  <Step title="Step 3: Create Customer">
    Navigate to **Flexprice Dashboard** → **Customer Management** → **Customers**


     <Frame>
  ![Create Customer in Flexprice](/public/images/docs/subscription-flow/credit-based-customer.png)
  </Frame>


    **Required:**
    - **Customer Name**: e.g. `Flex Corporation`

    For details, check-out [Customer Management](/docs/customers/create).
  </Step>

  <Step title="Step 4: Create Wallet in Credits">
    Open the customer you created → **Wallet** tab.

  <Frame>
  ![Create Wallet in Credits (CRD)](/public/images/docs/subscription-flow/credit-based-wallet.png)
  </Frame>

    **Wallet configuration:**
    - **Currency**: **Custom** → select your Price Unit (e.g. **Credits (CRD)**)
    - Balance will be shown and stored in credits; Flexprice converts to base currency for billing.

    For more, check-out [Create Wallet](/docs/wallet/create) and [Custom pricing use cases](/docs/product-catalogue/custom-pricing/use-cases).
  </Step>

  <Step title="Step 5: Credit Grant and Entitlement (Plan-Based Path)">
    For **plan-based credits:** Open the customer → **Add Subscription** → select the plan you created. When the subscription is active, the customer receives the plan’s **credit grants** (e.g. recurring monthly credits) and **entitlement** to the features linked to that plan. Entitlements define which features they can use and, if you configured it, free usage before deducting from the wallet.

    <Frame>
  ![Add Subscription and credit grants](/public/images/docs/subscription-flow/credit-grants.png)
  </Frame>

    For **direct purchase only**, skip this step and use Step 7 to top up the wallet. For more, check-out [Plans](/docs/product-catalogue/plans/create) and [Linking features to plans](/docs/product-catalogue/features/linking-to-plans).
  </Step>

  <Step title="Step 6: Top Up the Wallet (Initial or Add-On Credits)">
    On the customer’s **Wallet** tab, click **Top-up Wallet**.

    - Choose the credit type that matches your Price Unit (**Custom** → Credits).
    - Enter the amount in credits (e.g. 1,000).
    - Complete the top-up (e.g. **Add Credits** or link to payment).

    The wallet balance increases in credits. Use this for initial purchase or add-on packs.

    For more, check-out [Top-up wallet](/docs/wallet/top-up) and [Auto top-up](/docs/wallet/auto-top-up).
  </Step>

  <Step title="Step 7: Debit Wallet When User Performs an Action">
    When a user performs an action (search, export, enrichment, etc.) in your product:

    <Steps>
      <Step title="Check entitlement">
        **Check entitlement**: Does this customer’s plan allow this feature? Entitlements (and optional free usage before deducting credits) are configured when [linking features to plans](/docs/product-catalogue/features/linking-to-plans). If the customer has no access, block the action.
      </Step>
      <Step title="Look up credit cost">
        **Look up the credit cost** for that action (from your mapping, e.g. the credit cost tables in Section 1 (Key Chargeable Features in Apollo)).
      </Step>
      <Step title="Check balance">
        **Check** the customer’s wallet balance (via Flexprice API or dashboard).
      </Step>
      <Step title="Debit if sufficient">
        If **balance ≥ cost**: call the Flexprice API to **debit** the wallet by that amount, then perform the action.
      </Step>
      <Step title="Reject or prompt if insufficient">
        If **balance &lt; cost**: return an error or prompt to purchase more credits.
      </Step>
    </Steps>

    **Debit via API:** Use the Flexprice API to deduct from the wallet (see your API reference for wallet debit endpoint). Pass the customer identifier (e.g. External ID) and the amount in the **credit unit**.

    **Debit via Dashboard:** **Customer** → **Wallet** → **Manual Debit**; enter the amount in credits and reason (e.g. “Contact export – 5 credits”).

    For more, check-out [Manual Debit](/docs/wallet/manual-debit).
  </Step>

  <Step title="Step 8 (Optional): Auto Top-Up When Balance Is Low">
    To automatically add credits when the wallet balance falls below a threshold:

    Configure **Auto top-up** for the wallet (e.g. when balance &lt; 100 credits, charge the customer and add 1,000 credits). Flexprice supports configurable triggers and amounts.

    For more, check-out [Auto top-up](/docs/wallet/auto-top-up).
  </Step>

  <Step title="Step 9: Verify Balance and Transactions">
    <Steps>
      <Step title="Wallet balance">
        **Customer** → **Wallet** → view balance in credits.
      </Step>
      <Step title="Transaction history">
        **Wallet** → **Transactions** to see top-ups and debits.
      </Step>
      <Step title="Invoices">
        Billing and invoices use base currency; line items can show credit usage where applicable.
      </Step>
    </Steps>

    For more, check-out [Wallet transactions](/docs/wallet/transactions).
  </Step>
</Steps>

## Next Steps

After completing the setup:

<Steps>
  <Step title="Map actions to credit costs">
    Map actions to credit costs in your application (e.g. 1 credit per search, 5 per export) and call the debit API (or Manual Debit) after each action.
  </Step>
  <Step title="Monitor wallet balances">
    Set low-balance alerts if needed; see [Low balance alert](/docs/product-catalogue/features/wallet-balance-alert).
  </Step>
  <Step title="Review invoices">
    Check that credit top-ups and usage are reflected correctly in base currency.
  </Step>
  <Step title="Optional: Plans with credit packages">
    Create **plans** that include credit packages as recurring or one-time charges using the same Price Unit; see [Custom pricing use cases](/docs/product-catalogue/custom-pricing/use-cases).
  </Step>
</Steps>

For custom pricing concepts and API details, see [Custom Pricing](/docs/product-catalogue/custom-pricing/pricing-unit) and the [API Reference](/api-reference/introduction).
